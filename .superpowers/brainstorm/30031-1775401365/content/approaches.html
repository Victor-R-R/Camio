Architecture — 3 approches possibles

Même stack (Next.js 15 + Prisma + Supabase), différences dans l'organisation du code

A ⭐

Next.js monolithique — App Router + Server Actions

Tout dans une seule app Next.js. Les mutations (créer liste, cocher item) passent par des Server Actions. Les routes API sont réservées aux webhooks et au lien public magasinier.

Pourquoi c'est le bon choix

  • Zéro over-engineering pour une app mono-entreprise
  • Server Actions = typage end-to-end, pas de fetch/JSON à la main
  • Déploiement Vercel zero-config
  • Codebase compact, maintenable par une personne

Limites

  • Si besoin d'une API mobile native plus tard, il faudra ajouter des Route Handlers
B

Next.js + API REST séparée (Route Handlers uniquement)

Toutes les mutations passent par des endpoints REST (/api/lists, /api/items, etc.). Le frontend utilise fetch() ou SWR pour tout.

Avantages

  • API réutilisable si application mobile future
  • Séparation claire frontend/backend

Limites

  • Beaucoup plus de boilerplate (validation, sérialisation, fetch)
  • Inutile pour un usage mono-équipe sans app mobile prévue
C

Next.js + tRPC

API typée end-to-end via tRPC. Idéal pour les grosses équipes ou apps complexes.

Avantages

  • Typage parfait client-serveur
  • DX excellent pour les grosses équipes

Limites

  • Overkill pour cette taille d'app
  • Courbe d'apprentissage, configuration initiale lourde
  • Ajoute une dépendance majeure inutile ici

💡 Recommandation : Option A — Server Actions pour les mutations, Route Handlers uniquement pour le lien public magasinier (/api/check/[token]) et l'upload logo.
