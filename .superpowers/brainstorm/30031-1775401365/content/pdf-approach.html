Génération des 3 bons PDF — quelle approche ?

Ce choix impacte l'architecture, la qualité des PDFs et la complexité de déploiement

A

CSS @media print + window.print()

Les 3 bons sont des composants React mis en forme avec du CSS print. L'utilisateur clique "Imprimer" et le navigateur génère le PDF via la boîte de dialogue standard.

Avantages

  • Zéro dépendance externe
  • Rendu fidèle au HTML
  • Marche sur mobile (tablette magasinier)
  • CSS déjà prévu dans le cahier des charges

Limites

  • Pas de téléchargement direct .pdf
  • Mise en page dépend du navigateur
  • Moins de contrôle sur les sauts de page
B

react-pdf (client-side)

Les bons sont définis en JSX React-PDF, générés côté client comme de vrais fichiers PDF téléchargeables.

Avantages

  • Téléchargement .pdf direct
  • Contrôle précis de la mise en page
  • Portable, sans dépendance serveur

Limites

  • API différente du HTML/CSS (apprendre)
  • Rendu légèrement différent du HTML
  • Bundle JS plus lourd (~300KB)
C

Puppeteer / Playwright (server-side)

Un endpoint API génère le PDF côté serveur en rendant la page HTML dans un navigateur headless.

Avantages

  • Rendu pixel-perfect identique au HTML
  • PDF téléchargeable + envoyable par email
  • Indépendant du navigateur client

Limites

  • Complexe à déployer sur Vercel (taille image)
  • Coût en temps d'exécution (serverless)
  • Peut nécessiter un service externe (Browserless)

💡 Ma recommandation : Option A (CSS print) — tu as déjà prévu le CSS @media print dans ton cahier des charges, c'est le plus simple et parfaitement adapté à l'usage (impression bureau). On peut ajouter un bouton "Sauvegarder PDF" avec react-to-print si besoin.
